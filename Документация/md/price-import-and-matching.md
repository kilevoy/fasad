# Импорт Прайса И Универсальный Матчинг

Этот файл задает общий механизм, по которому новый прайс должен связываться
с нормализованными каталогами калькулятора.

Связанные файлы:
- [catalog-registry.seed.json](C:\КАЛЬКУЛЯТОР КАССЕТ\Документация\data\catalog-registry.seed.json)
- [price-matching.config.json](C:\КАЛЬКУЛЯТОР КАССЕТ\Документация\data\price-matching.config.json)
- [master-catalog-overview.md](C:\КАЛЬКУЛЯТОР КАССЕТ\Документация\md\master-catalog-overview.md)

## 1. Цель слоя импорта

Импорт нового Excel не должен ломать расчетную модель.

Задача слоя импорта:
- прочитать новый прайс;
- нормализовать строки;
- сопоставить их с seed-каталогами;
- обновить только цены, коды, веса и статусы;
- не трогать инженерные правила.

## 2. Общий пайплайн

### Шаг 1. Чтение исходного файла

Из прайса нужно извлекать как минимум:
- исходное название;
- код;
- группу;
- цену;
- единицу;
- вес.

### Шаг 2. Нормализация строки

Перед матчингом строка должна быть приведена к структурированным полям:
- семейство;
- роль;
- размер;
- толщина;
- покрытие;
- стандартная длина;
- марка стали;
- специальные признаки.

### Шаг 3. Определение каталога

Каждая строка должна попадать в один из каталогов:
- `cassettes`
- `corner_cassettes`
- `accessories_and_trims`
- `subsystem`
- `insulation_and_membrane`
- `fasteners`

### Шаг 4. Матчинг по ключам

Для каждого каталога использовать только его ключи из:
- [price-matching.config.json](C:\КАЛЬКУЛЯТОР КАССЕТ\Документация\data\price-matching.config.json)

### Шаг 5. Обновление seed

Если найдено точное совпадение:
- обновить `code`
- обновить `price`
- обновить `weight`
- обновить `status = active`

Если совпадение не найдено:
- не подставлять похожую строку вручную;
- оставить запись со статусом `needs_price`;
- создать запись в журнале расхождений.

## 3. Что именно обновляется

Импорт прайса имеет право менять только:
- `code`
- `price`
- `weight`
- `status`
- при необходимости `name`

Импорт прайса не должен автоматически менять:
- инженерные нормы;
- правила стандартности;
- швы;
- ограничения по размерам;
- нормы расхода крепежа.

## 4. Каталоги и ключи матчинга

### Основные кассеты

Ключ:
- `family + role + thickness + coating`

Пример:
- `kf1 + standard + 0.7 + polyester`

### Угловые кассеты

Ключ:
- `family + orientation + thickness + coating`

Пример:
- `ukf4_30 + vertical + 1.2 + colorflow_1s`

### Фасонные элементы

Ключ:
- `family + mark + standardLengthMm + coating`

Дополнительно при необходимости:
- `sizeVariant`
- `thickness`

### Подсистема

Ключ зависит от семейства:
- `kvgu`: `family + widthSeries + length + thickness + coating + steelGrade`
- `npp`: `family + width + height + thickness + coating`
- `npg`: `family + leg + thickness + coating`
- и так далее

### Утеплитель и мембрана

Сейчас это инженерный слой.
Значит при появлении прайса:
- матчить только на уже подготовленные семейства и режимы использования;
- не создавать новые инженерные категории автоматически.

### Крепеж

Ключ:
- `family + diameter + length + finish`

## 5. Запрет на опасное приближение

Главное правило:
- нельзя подставлять “похожую” строку, если нет точного совпадения по ключу.

Примеры ошибок, которых надо избегать:
- вместо `КВГУ 50x230 2.0 (Оцинк.)` брать окрашенный вариант;
- вместо `КФ4/17` брать `КФ4/30`;
- вместо `УКФ` брать `УГКФ`;
- вместо `НПШ 20x50x20` брать `НПШ20 0,7х3м`.

## 6. Журнал расхождений

Для надежной работы нужен отдельный журнал импорта.

Минимальный формат записи:

```json
{
  "sourceName": "Кассета фасадная 4/30 (КФ4/30) 1,2 (Колор-поток)",
  "catalogId": "cassettes",
  "normalizedCandidate": {
    "family": "kf4_30",
    "role": "standard",
    "thickness": 1.2,
    "coating": "colorflow_1s"
  },
  "matchStatus": "matched | unmatched | ambiguous",
  "matchedSeedId": null,
  "notes": "не найдено точное совпадение"
}
```

## 7. Рекомендуемая структура выходов импорта

После обработки прайса логично формировать три результата:

1. `matched`
   - строки, которые точно сопоставились

2. `unmatched`
   - строки, для которых нет подходящей seed-записи

3. `missing_prices`
   - seed-записи, которые все еще не нашли цену в новом прайсе

## 8. Минимальный практический результат для проекта

После реализации этого слоя мы сможем:
- обновлять прайс без ручного переписывания всех каталогов;
- видеть, какие позиции исчезли из прайса;
- видеть, какие новые позиции появились;
- безопасно доращивать каталоги без хаоса.
